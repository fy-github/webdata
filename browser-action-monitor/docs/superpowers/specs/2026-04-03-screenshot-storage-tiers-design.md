# Screenshot Storage Tiers Design

Date: 2026-04-03

## 背景

当前扩展已经支持为动作附加截图与 DOM 快照，但截图能力在真实使用中很容易被自动关闭。

现状的主要问题是：

- 截图默认总存储上限仅为 `8 MB`
- 截图内容当前以 `dataUrl` 形式存入扩展本地存储，体积膨胀明显
- 存储压力一旦达到阈值，截图能力会自动关闭
- 用户的目标是“保数据”，不能通过自动删除历史截图来换取空间

同时，用户明确要求：

- 本地和服务端都长期保留完整截图
- 仍然保留“达到上限后自动关闭截图”的保护行为
- 不做历史数据自动删除
- 设置中不要暴露零散参数，而是用可选档位管理截图策略

## 目标

- 将截图存储能力升级为档位化配置，而不是固定硬编码阈值
- 让每个档位同时控制：
  - 本地截图总存储上限
  - 截图压缩质量
  - 最大截图宽度
- 把截图字节从当前的扩展状态存储迁移到 `IndexedDB`
- 保留本地与服务端的双份截图副本
- 保留“达到上限后自动关闭截图”的保护逻辑
- 支持用户切换到更高档位后自动恢复截图能力

## 非目标

- 本期不做“本地绝对路径可配置”
- 不引入本地 companion 服务
- 不做上传成功后的本地截图清理
- 不做历史截图的全量强制迁移
- 不新增复杂的手动质量、尺寸、限速细粒度调参界面

## 已确认的产品方向

- 采用三档可选策略，而不是写死单个 MB 值
- 三档必须联动质量与分辨率，不能只是改存储上限
- 达到所选档位上限后，截图自动关闭
- 已保存的本地截图和服务端截图都必须保留
- 当用户切换到更高档位时，如果空间重新满足条件，截图自动恢复
- “手动关闭”与“因上限自动关闭”必须严格区分

## 档位设计

### 轻量

- 本地总上限：`64 MB`
- JPEG 质量：`0.55`
- 最大宽度：`1280`

适用场景：

- 日常记录
- 长时间挂录
- 优先降低浏览器压力

### 平衡

- 本地总上限：`256 MB`
- JPEG 质量：`0.72`
- 最大宽度：`1600`

适用场景：

- 默认推荐档位
- 在容量、质量、稳定性之间取得折中

### 重度取证

- 本地总上限：`1024 MB`
- JPEG 质量：`0.86`
- 最大宽度：`1920`

适用场景：

- 更强调证据保留
- 接受更高磁盘占用和更高本地压力

## 高层架构

本期采用“元数据继续走现有状态存储，截图字节单独走 `IndexedDB`”的方案。

核心拆分如下：

1. 动作记录与会话状态继续走现有本地状态存储
2. 动作中的截图字段只保留元数据，不再内嵌大体积 `dataUrl`
3. 截图实际字节内容写入 `IndexedDB`
4. 本地预览、导出、上传都通过统一的截图读取层访问截图字节
5. 服务端继续保留远端副本，不参与本地空间回收

这个边界保证：

- 现有动作模型改动可控
- records、popup、sync、server 的上层逻辑大部分仍可复用
- 真正重的存储压力被从 `chrome.storage.local` 中移走

## 数据模型

### 设置模型

新增一个截图存储档位字段，例如：

- `screenshotStorageProfile`

可选值：

- `light`
- `balanced`
- `forensics`

该字段是档位入口，其他联动参数由内部档位表统一推导，不在设置页暴露为独立输入框。

### 档位配置表

在扩展共享配置中集中维护截图档位表，每档包含：

- `storageLimitMb`
- `jpegQuality`
- `maxWidth`

上层只读取档位，不直接散落读取多个魔法数。

### 动作截图元数据

动作中的 `attachment.screenshot` 继续保留现有元数据结构，例如：

- `status`
- `storageKey`
- `mimeType`
- `width`
- `height`
- `capturedAt`
- `error`
- `remoteStatus`
- `remoteUrl`
- `remoteUploadedAt`
- `remoteError`

变化点仅在于：

- `storageKey` 指向 `IndexedDB` 内部记录，而不是依赖旧的 `chrome.storage.local` 二进制载荷

## 存储设计

### 本地存储

截图字节改为写入 `IndexedDB`，不再把完整截图 `dataUrl` 放入动作状态存储。

`IndexedDB` 中每条记录至少包含：

- `actionId`
- `sessionId`
- `blob` 或等价二进制内容
- `mimeType`
- `width`
- `height`
- `sizeBytes`
- `savedAt`

### 本地占用统计

截图健康状态中的 `screenshotStorageBytes` 继续保留，但其来源改为：

- 来自 `IndexedDB` 存储写入后的字节统计
- 而不是 `dataUrl` 本地状态估算

### 服务端保留

服务端继续保留截图第二份副本：

- 上传成功后不删除本地截图
- 本地与服务端相互独立
- 本期不做副本协调清理

## 兼容策略

为了避免升级时丢失历史截图，本期采用“新写入走新存储，读取兼容旧存储”的方式。

规则如下：

- 新截图统一写入 `IndexedDB`
- 读取截图时先查新存储
- 如果新存储不存在，再回退到旧存储路径
- 不执行全量历史迁移任务

这样可以：

- 降低升级风险
- 避免一次性迁移大量截图导致浏览器卡顿或失败
- 保持已有历史记录仍可访问

## 自动关闭与自动恢复

### 关闭原因区分

必须明确区分以下两类状态：

- `disabled_manual`
- `disabled_auto_quota`

只有 `disabled_auto_quota` 才允许被“切换到更高档位”自动恢复。

### 达到上限时

当截图总占用达到当前档位上限时：

- 停止采集新截图
- 截图功能自动关闭
- 关闭原因标记为 `storage-pressure`
- 已保存截图保留
- 服务端副本保留

### 升档恢复

当用户从低档位切换到更高档位时：

1. 重新读取当前本地截图占用
2. 对比新档位上限
3. 若当前占用低于新上限：
   - 自动恢复截图能力
   - 清除自动关闭原因
4. 若仍高于新上限：
   - 保持自动关闭

### 手动关闭优先级

如果用户手动关闭截图：

- 切换档位不应自动重新开启
- 只有用户自己重新开启才恢复采集

## UI 设计

### 设置页

在 `Browser Action Monitor 设置` 中新增“截图存储档位”配置项。

展示形式建议为单选式分组或下拉选项，文案为：

- `轻量`
- `平衡（推荐）`
- `重度取证`

每个选项同时展示简短说明，例如：

- 轻量：64 MB，质量较低，最稳
- 平衡：256 MB，推荐默认
- 重度取证：1024 MB，更高保真

### Popup

Popup 继续显示：

- 截图是否开启
- 当前本地占用

本期可选增强：

- 增加当前档位名称显示

但不是必须项。

### Records 页面

records 读取截图预览时改为通过统一截图读取层访问：

- 优先读 `IndexedDB`
- 回退读旧存储

界面本身不需要大改。

## 模块边界

### 需要修改的模块

- `manifest.json`
  - 增加 `unlimitedStorage`
- `src/extension/shared/constants.js`
  - 增加截图档位默认值与配置表
- `src/extension/background/attachment-store.js`
  - 新增 `IndexedDB` 适配器
  - 保留统一存取接口
- `src/extension/background/attachment-queue.js`
  - 写入截图时接入新存储
  - 统计真实占用
- `src/extension/background/screenshot-policy.js`
  - 从档位表读取存储上限
  - 支持升档自动恢复规则
- `src/ui/options.js`
  - 加载与保存截图档位设置
- `options.html`
  - 新增截图档位配置项
- `src/ui/records.js`
  - 通过统一读取层获取截图
- 可能涉及导出与同步相关模块
  - 确保截图字节读取不依赖旧存储路径

### 建议新增模块

- `IndexedDB` 截图存储适配器
- 截图档位解析 helper
- 可选的截图编码参数 helper

## 风险与约束

### 1. 浏览器磁盘占用上升

这是接受 `unlimitedStorage + IndexedDB + 双份保留` 的直接结果，不属于 bug，而是产品选择。

### 2. 读取链路兼容复杂度增加

因为本期不做全量迁移，所以读取层需要兼容：

- 新存储
- 旧存储

### 3. 状态一致性

本地截图、动作元数据、服务端上传状态三者分离后，需要保证：

- 本地截图存在但远端未上传
- 远端已上传但本地仍保留
- 历史记录仍可回退读取

这些状态都必须能被 UI 正确表达。

## 测试策略

本期至少覆盖：

- 档位配置表解析
- 默认档位加载与保存
- 不同档位下的上限计算
- 达到上限后自动关闭截图
- 切换到更高档位后自动恢复截图
- 手动关闭状态不会被升档自动恢复
- `IndexedDB` 写入、读取、删除、统计
- 新写入走新存储
- 历史数据读取可回退旧存储
- records / popup 对新状态的展示
- 全量回归 `npm test`

## 推荐结论

本期采用“档位化截图策略 + `IndexedDB` 本地存储 + `unlimitedStorage` + 双份保留 + 到上限自动关闭、升档自动恢复”的组合方案。

这是当前最平衡的路线：

- 它解决了“几个页面就自动关闭”的核心容量问题
- 它保留了用户明确要求的双份数据副本
- 它避免把系统一下拉进“本地磁盘绝对路径可配置”的高复杂度阶段
- 它把变化集中在截图存储层和策略层，不需要推倒现有动作模型
